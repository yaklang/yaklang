Global HotPatch 负责动态取会话、补 `X-Pipeline-*` 头，以及把响应解回明文 JSON。
Module HotPatch 继续盯住业务请求体，直接改 `keyword`，做 SQL 注入 payload 变异和命中判断。
为什么不是前端加密案例 因为这里故意让请求体保持明文，重点讲 pipeline 的分层，而不是讲前端 JS 逆向。

为什么这个靶场适合讲全局热加载

直接去实操

如果一个场景只是“前端自己先 AES 加密，再把密文交给服务端”，那更像是协议复刻问题， 模块热加载自己就能搞定，未必能把 global 的价值讲出来。这个靶场故意把协议层和业务层拆开，让 pipeline 更清楚。

  1. 请求体保持明文 JSON，模块热加载还能直接改 `keyword`、`status`、`page` 这些业务字段。
  2. 请求之前必须先去拿动态会话，并补 `X-Pipeline-Session`、`X-Pipeline-Timestamp`、`X-Pipeline-Signature`，这部分天然适合放在全局热加载。
  3. 服务端统一返回 AES-CBC 信封，响应回来先由全局热加载解包，模块热加载再根据明文结果做命中判断。
签名公式 hex(HMAC-SHA256(session_key, METHOD + "\n" + PATH + "\n" + TIMESTAMP))

协议摘要

步骤 1 `GET {{.BootstrapPath}}`，拿到 `ticket` 和 `iv`。
步骤 2 用固定引导密钥 `{{.BootstrapKey}}` 解密 `ticket`，得到 `session_id` 和 `session_key`。
步骤 3 用明文 JSON 请求 `POST {{.SearchPath}}`，只在 Header 里带动态签名。
步骤 4 如果响应头里有 `X-Pipeline-Encrypted: 1`，就继续用 `session_key` 解 `data` 字段。

对应到全局热加载时，应该怎么拆

  • `beforeRequest(global)`：先请求 bootstrap，缓存 `session_id`、`session_key`，再补齐 `X-Pipeline-*` 头。
  • `beforeRequest(module)`：继续改业务参数，把 `keyword` 替换成注入 payload。
  • `afterResponse(global)`：识别 `X-Pipeline-Encrypted: 1`，把 `data` 解成明文 JSON。
  • `afterResponse(module)`：基于明文响应判断是否命中，例如 `row_count` 是否暴涨、是否出现 `admin` 结果。
