## 输出策略

- 最终默认只需要产出一个高质量的 `task_long_summary`，它会作为主总结对外展示。
- `status_summary` 是兼容字段，只有在存在无法丢失的执行中状态时才填写；如果会和最终总结或 next movements 重复，就留空。
- `task_short_summary` 也是兼容字段。只有在你能给出一条明显更短、且对任务树展示更有价值的单句摘要时才填写；否则留空。
- 不要把同一信息拆成三段重复表达。宁可把证据、结论和建议集中写进 `task_long_summary`。

## status_summary 字段输出要求（可选）

如果填写 `status_summary`，只保留本轮尚未被主总结覆盖的中间态，例如“已完成基础枚举，待核实异常连接是否真实可疑”。不要长篇展开。

## task_long_summary 字段输出要求与示例

`task_long_summary` 是本次任务完成后的主总结，必须优先保证质量。请尽量用 3-6 句写清楚：

1. 你做了什么操作，或用了哪些关键工具/命令。
2. 你拿到了哪些关键证据、数值、配置或事实。
3. 你的核心发现或结论是什么。
4. 如有必要，再补一句后续建议。

要求：

- 信息密度高，但不要写成长报告。
- 证据必须具体，不要只写“已完成检查”“结果正常”这类空话。
- 不要重复 timeline 中的细枝末节，只保留关键证据。
- 不要再额外复述一版简短总结，除非确实需要填写 `task_short_summary`。

**示例 1 - 系统检查类任务**:
```
系统配置检查完成。执行了 uname -a、hostname、uptime 采集基础信息。
关键数据：CentOS 7.9 x86_64，内核 3.10.0-1160，运行 127 天，CPU 8 核，内存 16GB 已用 78%，根分区使用率 65%。
发现内存使用率偏高，但基础配置整体正常。
建议下一步定位高内存进程，并评估维护窗口内重启需求。
```

**示例 2 - 网络检查类任务**:
```
网络连接检查完成。使用 ss -tulnp 和 netstat 扫描 TCP/UDP 连接状态。
关键数据：共 23 个监听端口，对外开放 22、80、443，MySQL 仅监听 127.0.0.1；活跃连接 156 个，其中 ESTABLISHED 89 个。
安全发现：5 次境外 SSH 尝试已被 fail2ban 拦截。
结论：当前网络暴露面基本可控，后续应持续复核 SSH 防护日志与访问控制策略。
```

**示例 3 - 安全扫描类任务**:
```
Web 服务安全扫描完成，目标为 Nginx 1.18.0 和 Tomcat 8.5.65。
关键证据：Tomcat 管理界面对外暴露，Nginx 未关闭 server_tokens，TLS 仍包含弱套件 TLS_RSA_WITH_AES_128_CBC_SHA。
风险结论：当前存在 1 个中危入口和若干低危配置项，无高危漏洞。
建议优先限制 Tomcat 管理界面访问，其次收紧 Nginx 信息泄露与 TLS 配置。
```

## task_short_summary 字段输出要求（可选）

只有在它比 `task_long_summary` 明显更短、且能独立承担任务树展示职责时才填写。例如：

```
已确认目标为 Nginx，后续改查配置。
```
