## 输出策略

- 最终默认只需要产出一个高质量的 `task_long_summary`，它会作为主总结对外展示。
- `status_summary` 是兼容字段，只有在存在无法丢失的执行中状态时才填写；如果会和最终总结或 next movements 重复，就留空。
- `task_short_summary` 也是兼容字段。只有在你能给出一条明显更短、且对任务树展示更有价值的单句摘要时才填写；否则留空。
- 不要把同一信息拆成三段重复表达。宁可把证据、交付结果、结论和建议集中写进 `task_long_summary`。
- 默认优先总结“交付了什么”“确认了什么”“遗留了什么”，而不是按时间顺序复述“先做了什么、再做了什么”。
- 可以充分使用 Markdown 结构化表达，但必须简洁，避免写成长报告。
- `task_long_summary` 会和 EVIDENCE 一样进入共享执行上下文，供后续任务继续引用；因此必须写成可交接的结果摘要，而不是过程流水账。

## status_summary 字段输出要求（可选）

如果填写 `status_summary`，只保留本轮尚未被主总结覆盖的中间态，例如“已完成基础枚举，待核实异常连接是否真实可疑”。不要长篇展开。

## task_long_summary 字段输出要求与示例

`task_long_summary` 是本次任务完成后的主总结，必须优先保证质量。默认写成“结果导向”的 Markdown，总结后续接手者最需要的信息。

优先采用下面的结构：

```markdown
### 关键结果
- 列出已确认的关键事实、数值、配置、暴露面、文件、端口、服务、风险点或产物。
- 只保留后续决策需要引用的信息。

### 结论 / 风险
- 提炼核心判断：当前是否正常、异常点在哪里、影响面如何。
```

如果任务很小，也可以压缩成 3-6 行短段落，但仍然要遵守“结果优先、过程靠后”的原则。

要求：

- 信息密度高，但不要写成长报告。
- 证据必须具体，不要只写“已完成检查”“结果正常”这类空话。
- 不要重复 timeline 中的细枝末节，只保留关键证据。
- 不要把脚本、命令、工具调用过程按顺序重述一遍，除非某个命令本身就是关键证据来源。
- 优先写结果、事实、结论、产物；过程只保留 1 句以内背景说明。
- 不要再额外复述一版简短总结，除非确实需要填写 `task_short_summary`。
- 如果存在明确交付物，优先写清楚：文件、报告、配置结论、风险判断、可复核数据。
- 只有当文件路径在时间线、`output_files`、工具 stdout 或文件写入记录中出现过时，才能把它写成“已生成交付物”；如果只是 AI 在回答中提到某个路径，但没有工具证据，不得写成既成事实。
- 对执行型安全任务，只有真实请求、响应差异、浏览器执行、插件结果或证据文件，才能写成“已验证漏洞”；技能加载、Payload 清单、测试计划只能写成“已制定计划”或“待验证”。
- 不要再额外维护“可继承事实”或“可继承结果”小节。凡是需要后续任务复用的执行证据，应通过 verification 的 `evidence` 字段或 `output_evidence` 动作写入共享 EVIDENCE 文档，由系统统一传播。

反例：

- 差："先执行 ps，再执行 ss，然后执行 lsof，最后分析结果。"
- 差："通过多条命令进行了全面检查，发现若干问题。"
- 好："已确认 26 个监听端口、30+ 个活跃外连、2 条 VPN 隧道与 1 个公网暴露的 Java 监听面；防火墙关闭，需优先复核暴露服务是否符合预期。"

**示例 1 - 系统检查类任务**:
```
### 关键结果
- 操作系统为 CentOS 7.9 x86_64，内核 3.10.0-1160，已连续运行 127 天。
- CPU 8 核，内存 16GB 已用 78%，根分区使用率 65%。

### 结论 / 风险
- 基础配置无异常，但内存接近高位，存在持续上涨后触发性能抖动的风险。
```

**示例 2 - 网络检查类任务**:
```
### 关键结果
- 共 23 个监听端口，对外开放的核心端口为 22、80、443。
- MySQL 仅监听 127.0.0.1；当前活跃连接 156 个，其中 ESTABLISHED 89 个。
- fail2ban 已拦截 5 次境外 SSH 尝试。

### 结论 / 风险
- 当前未见明显失控暴露，但 SSH 面持续受到探测，访问控制和防护日志需要继续盯防。
```

**示例 3 - 安全扫描类任务**:
```
### 关键结果
- 目标服务为 Nginx 1.18.0 和 Tomcat 8.5.65。
- Tomcat 管理界面对外暴露；Nginx 未关闭 server_tokens；TLS 仍包含弱套件 `TLS_RSA_WITH_AES_128_CBC_SHA`。

### 结论 / 风险
- 当前不存在高危漏洞，但管理界面暴露可直接扩大攻击面，应视为优先整改项。
```

**示例 4 - 主机网络状态巡检类任务**:
```
### 关键结果
- 系统存在 26 个 TCP 监听端口，主要服务包括 WeChat、ControlCenter、多实例 Chromium 与 Java 进程。
- 活跃 ESTABLISHED 连接超过 30 个，主要外连涉及 117.186.219.114:13269、1.3.107.246:443、1.1.2.121:443。
- 识别到 2 条 VPN 隧道 `utun6` 与 `utun8`，DNS 使用 223.6.6.6。
- 系统防火墙处于关闭状态，未看到异常外连直连可疑公网 IP，但 Java 监听端口存在公网暴露风险。

### 结论 / 风险
- 当前更像是“存在较大暴露面但未见直接入侵证据”的状态；风险点在于防火墙关闭、监听面较多、部分外连目标需确认业务归属。
```

## task_short_summary 字段输出要求（可选）

只有在它比 `task_long_summary` 明显更短、且能独立承担任务树展示职责时才填写。例如：

```
已确认目标为 Nginx，后续改查配置。
```
