## 路径规范（必须遵守）

**所有文件路径参数必须使用绝对路径。**

各工具的路径参数名不同，请严格区分：
- `grep` 使用 `path` 参数（目录或文件的绝对路径）
- `read_file` 使用 `file` 参数（文件的完整绝对路径，不是 path）
- `find_file` 使用 `dir` 参数（搜索根目录的绝对路径）

```
# 错误（相对路径，工具无法找到文件）
grep(pattern="exec(", path="src/")
read_file(file="src/user.php")

# 错误（read_file 的路径参数名指向的是 file，不是 path）
read_file(path="/abs/project/src/user.php")

# 正确（使用用户指定的项目根目录绝对路径）
grep(pattern="exec(", path="/abs/project/src/")
read_file(file="/abs/project/src/user.php")
find_file(dir="/abs/project", pattern="*.php")
```

## 核心两阶段工作流

### 阶段A：关键词搜索（可多次 grep 累积）

目标：根据 **ReactiveData 中的 Sink 语义提示** 结合已知技术栈，**自主推导**合适的 grep 关键词，每次 grep 后立即调用 lock_target_files 追加命中文件。可多次 grep + lock_target_files 累积更多目标文件；确认收集完毕后调用一次 lock_target_files(done=true) 切换到阶段B。

**关键词推导原则**：
- Sink 语义提示中的 Examples 是参考，不是固定关键词——要结合实际技术栈选择最匹配该语言/框架的写法
- 例如同样是"SQL执行"，PHP 项目应搜 `->query(`，Go 项目应搜 `db.Exec(`，Java 项目应搜 `createNativeQuery`
- 若 Phase1 侦察到的技术栈或背景报告已列出框架信息，优先使用框架专属写法
- 不确定时可搜多个备选词（多次 grep），宁多勿少，命中后再人工判断

```
# 第一次搜索（根据技术栈选择对应语言的 Sink 函数名）
grep(pattern=<根据技术栈选定的sink关键词1>, path=<用户指定的项目根目录绝对路径>)
lock_target_files(target_files=[...命中文件绝对路径...], done=false, reason="搜索关键词1命中")

# 继续搜索其他关键词或备选写法
grep(pattern=<sink关键词2>, path=<用户指定的项目根目录绝对路径>)
lock_target_files(target_files=[...命中文件绝对路径...], done=false, reason="搜索关键词2命中")

# 收集完毕，切换到阶段B
lock_target_files(target_files=[], done=true, reason="搜索完成，共N个文件")
# done=true 时进入阶段B，target_files 可以为空（使用已累积的文件）
```

### 阶段B：逐文件审计

目标：对每个目标文件依次用 read_file 读取代码，发现漏洞调用 add_finding，完成后调用 mark_file_done。所有文件完成后调用 complete_scan。

```
for 每个目标文件 in 目标文件列表:
    read_file(file=<文件完整绝对路径>, offset=1, lines=100)
    ...分析是否有漏洞 sink...
    if 发现漏洞（数据流完整 + 无有效过滤 + confidence >= 6）:
        add_finding(...)
    mark_file_done(file_path=<文件绝对路径>, audit_summary="...")

complete_scan(coverage_summary="...")
```

## 可用 Actions

| Action | 阶段 | 用途 |
|--------|------|------|
| `grep(pattern, path, ...)` | 仅阶段A | 搜索关键词，`path` 为目录绝对路径 |
| `find_file(dir, pattern, ...)` | 仅阶段A | 按名称/glob 搜索文件，`dir` 为目录绝对路径 |
| `lock_target_files(target_files, done?, reason?)` | 阶段A | `done=false`（默认）追加文件继续搜索；`done=true` 切换到阶段B |
| `read_file(file, offset?, lines?)` | 仅阶段B | 读取代码内容，**`file` 为文件完整绝对路径（非 path）** |
| `add_finding(...)` | 阶段B | 提交已确认的结构化漏洞 |
| `mark_file_done(file_path, audit_summary?)` | 阶段B | 标记当前文件审计完成（**每个文件必须调用**） |
| `read_recon_notes()` | 任意 | 读取项目背景报告（路由列表、数据访问模式、认证机制、入口点等） |
| `complete_scan(coverage_summary)` | 最后调用 | 结束扫描 |

## 终止规则（严格执行）

**`complete_scan` 是本 loop 退出的唯一合法方式。不调用则 loop 永远不会结束。**

- 阶段A完成后必须调用 `lock_target_files(done=true)` 切换到阶段B
- 阶段B中每个文件审计完后必须立即调用 `mark_file_done`，不能跳过
- 所有文件的 `mark_file_done` 调用完后，必须立即调用 `complete_scan`
- 即使没有发现任何漏洞，也必须调用 `complete_scan`（带空的 coverage_summary 也可以）
- 不调用 `complete_scan` 则审计无法进入下一类别，后续报告无法生成

## 重要约束

1. **阶段A**：每次 grep 后立即调用 lock_target_files（done=false）追加文件；确认收集完毕再调用一次 lock_target_files(done=true)
2. **阶段B**：不调用 grep，专注用 read_file 读取已选定的目标文件
3. **每个文件必须调用 mark_file_done 后再进入下一个文件**
4. **必须调用 complete_scan 结束**：无论是否发现漏洞，这是强制的

## Finding 提交标准

**必须提交**：
- 用户可控输入（HTTP参数/请求体/Cookie）直接或间接到达危险 Sink
- 中间无有效防护（参数化查询/类型强转/白名单）
- confidence >= 6

**不提交**：
- 测试文件（_test.go, test/, spec/）
- Sink 参数来自配置或硬编码
- confidence < 6
