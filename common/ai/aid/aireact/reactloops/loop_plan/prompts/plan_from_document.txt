# Output Requirements

## Plan Structure

- main_task: A concise name describing the overall task
- main_task_goal: A clear statement of what the entire plan achieves
- main_task_identifier: English snake_case semantic identifier for the main task
- tasks: An array of subtasks, each with the fields described below

## Subtask Fields

Each subtask MUST contain:

- subtask_name: A concise action-oriented name (use tool categories or method descriptions, NEVER write internal tool function names)
- subtask_identifier: English snake_case semantic identifier for directory naming
- subtask_goal: A DETAILED natural-language description that covers three aspects in a single paragraph (do NOT use section headers like "Objective:", "Acceptance Criteria:", "Completion Condition:" — just write directly):
  1. What specifically to do — list EVERY exact target, parameter, file path, URL, or configuration involved. If FACTS lists N targets, reference ALL N targets, not a subset.
  2. How to verify — specify concrete test cases/inputs (specific Payload, parameter value, request body), expected responses (HTTP status code, response body keyword, timing difference), and pass/fail rules (e.g., "if response contains 'syntax error' near quote character, SQL injection confirmed").
  3. When it is done — enumerate ALL test points that must be verified. The subtask is done when all points have a definitive pass/fail result.
- depends_on: Array of subtask_name values this task depends on (empty array if no dependencies)

## Hierarchical Task Structure (2-Level Subtasks)

Subtasks can contain `sub_subtasks` for finer-grained decomposition:
- Use sub_subtasks when a subtask naturally decomposes into 2 or more distinct steps targeting different specific inputs
- sub_subtasks follow the SAME field requirements as subtasks (subtask_name, subtask_identifier, subtask_goal, depends_on)
- Maximum nesting depth: 2 levels (subtask -> sub_subtask). Do NOT nest deeper.
- sub_subtasks inherit their parent subtask's depends_on context
- When sub_subtasks are present, the parent subtask_goal should be a high-level summary; detailed verification criteria go into each sub_subtask

## Subtask Granularity

- Each subtask (or sub_subtask at leaf level) should focus on ONE specific target or verification dimension
- The number of subtasks MUST be proportional to the number of distinct targets/endpoints/parameters discovered in FACTS
- Do NOT merge unrelated targets into a single subtask — if FACTS lists 3 SQL injection test points, create 3 sub_subtasks (one per test point), not 1 combined subtask
- If a single target requires multiple distinct verification steps (e.g., different payload types), use sub_subtasks to decompose

## Coverage Completeness (Critical)

### Single Source of Truth

**FACTS is the ONLY source of truth for coverage verification.** The Guidance Document does NOT enumerate subject lists (by design — it only declares methodology and acceptance rules), and it MAY contain abstractions or summaries that can miss items. **DO NOT** use the Guidance Document as your coverage checklist. **DO** read the FACTS block above line by line as your authoritative checklist.

### Coverage Rules

The plan MUST cover ALL subjects (targets / endpoints / files / modules / records / configuration items / any other form of "item to be processed") discovered in FACTS:
- Count every distinct subject bullet mentioned in FACTS, regardless of its concrete form (URL, file path, function/module name, record, configuration key, component, interface contract, etc.)
- Ensure EVERY single bullet appears in at least one subtask's or sub_subtask's goal, referenced by its exact name / path / identifier
- If a FACTS bullet is omitted from the plan, the plan is INCOMPLETE and MUST be expanded before returning
- **N-to-N hard quantification**: if FACTS lists N subjects, the union of all subtask_goal / sub_subtask subtask_goal strings MUST explicitly reference all N subjects by their exact identifiers. Representative sampling is forbidden — every item in a list-style FACTS block (URL list, file list, table list, directory list, etc.) must be named
- **Mandatory verification pass before returning**: re-scan FACTS line by line; for each bullet, confirm it is literally quoted (or its exact identifier is embedded) in at least one subtask_goal. Any missing subject means the plan is incomplete — expand sub_subtasks until coverage is complete
- For security testing: each (subject x vulnerability_type) combination that FACTS data suggests is testable MUST have a corresponding subtask or sub_subtask
- For development tasks: each file / module / component mentioned in FACTS that requires changes MUST appear in a subtask

## Summary Task Requirement

- When the plan involves security testing, vulnerability verification, or penetration testing, the LAST subtask in the plan SHOULD be a summary task that reviews all EVIDENCE from previous subtasks, consolidates confirmed vulnerability findings, and outputs structured risk records via `cybersecurity-risk` tool where applicable.
- This summary task depends on all preceding verification subtasks.

## Strict Prohibitions

- NEVER use vague words: "etc.", "and so on", "several", "some", "related", "similar", "various", "other", "and more" or their equivalents in any language (Chinese: "等", "其他", "若干", "一些", "相关", "类似", "各种")
- NEVER write generic goals like "test for vulnerabilities" or "verify security" — specify EXACTLY which vulnerability type, which endpoint, which parameter
- NEVER fabricate information not present in the guidance document, FACTS, or context above
- Every claim in subtask_goal must be traceable to a specific fact or document section
- NEVER collapse multiple distinct test points into one vague verification item

## Dependency Rules

- Use depends_on to express execution order
- Tasks with no dependencies can run in parallel
- Avoid unnecessary sequential dependencies — maximize parallelism where safe
