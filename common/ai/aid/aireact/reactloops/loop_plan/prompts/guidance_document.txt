# 输出要求

请按照以下**四章**结构输出一篇 Markdown 格式的任务执行指导性文档。每一章都必须扎根于上方的 FACTS 与 EVIDENCE，禁止编造上下文中不存在的信息。

## 通用写作约束（必读）

- **禁止复述 FACTS / EVIDENCE 的主体清单**：FACTS 与 EVIDENCE 是**唯一真相源**，下游 PLAN 将直接对照 FACTS 原文逐条核验覆盖率。本文档**不得**以任何形式（列表、枚举、段落罗列、表格）整批重新列出 FACTS 中的 URL 清单、文件清单、端点清单、模块清单、目录清单、记录清单等。重列不仅是浪费 token，更会因概括（出现"等"）导致下游把文档内清单当成权威而遗漏 FACTS 中的真实条目。
- **允许点名个别重点对象作为示例**：在"目标定义 / 可检验假设 / 执行路径 / 有效性"等章节中，为了让方法论落地更清晰，**可以**点名 1-3 个代表性对象作为举例（例如"以 /user/id 这类接口为代表的数字型输入点"）。但**禁止**把某一类下的全部对象铺开成清单，更禁止出现"包括 A、B、C 等"这种形式。
- **文档内同样严禁任何概括词**：不允许出现 "等"、"其他"、"相关"、"类似"、"若干"、"一些"、"多个"、"若干个"、"数个"、"部分"、"主要"，以及对应英文（etc. / and more / and so on / various / several / others / similar / including…）。如果一句话需要靠"等"来收尾，说明你在复述清单 — 立刻改写，要么只列举示例（"例如 X、Y"，不带"等"），要么完全不列举，用更抽象的方法论表述代替。
- **执行路径一章要克制细节**：执行路径的目的是给出"方法论骨架与方向"。繁琐的逐项枚举、逐条输入样例、代码片段、参数表留给下游 PLAN 阶段的子任务去展开。
- **保持领域无关的结构**：章节结构、小节名称、表述框架保持通用，不要以安全场景的专用词作为骨架；章节内的具体内容可以贴合本次任务的真实领域。
- 若某一章节在当前任务下确实无内容可写，保留章节标题并写一句"当前任务无相关内容"，而非编造。

## 文档结构（四章）

### 1. 目标定义 (Goal Definition)

必须包含以下**五要素**，每项独立小节，要写得具体、可核对：

- **目标内容**：精确重述用户原始需求，明确最终交付物是什么（一份报告？一组变更？一套结论？）
- **具体问题**：本次任务要回答或解决的核心问题是什么？应能用一两句话落到当前情境上
- **可检验假设**：列出若干可以通过实际执行来证实或证伪的假设，允许点名具体对象或具体条件（例如"目标 A 在输入 X 下会产生行为 Y"）；假设应可观测、可复现
- **预期方向**：解决该问题的主方向、优先级排序、为什么是这些方向
- **执行边界**：明确范围内 / 范围外；列出关键风险、约束条件、伦理与合规底线

### 2. 执行路径（方法论视角） (Execution Path — Methodology)

本章以**方法论**为核心，回答"这个任务属于什么领域 / 要解决的子问题是什么 / 应采用哪些方法论或检验范式"。要求：

- **领域识别**：先说明本任务所属的领域或交叉领域（尽量贴合真实场景，而非泛泛分类）
- **方法论主轴**：按"子问题 → 采用的方法论范式 / 检验思路"组织为若干条主轴。每条主轴说明：
  - 此范式要解决的子问题是什么
  - 采用哪种方法论、检验思路或常用工作流
  - 期望从中看到什么类别的信号 / 证据
- **避免沦为操作手册**：不要把每个具体对象的完整操作、参数、输入样例都塞进本章。逐项枚举、逐条输入/输出、具体脚本与命令留给下游 PLAN 阶段的子任务展开。本章可以以少量示例点明方法论如何落到当前任务上，但不是"对每个对象写一段步骤"。
- **禁止铺开 FACTS 清单**：本章点名具体对象**只**用于说明方法论范式（比如用 1-2 个代表性对象解释某条主轴适用于"此类输入"），**不得**以任何形式（bullet、表格、段落）铺开 FACTS 中同类主体的完整清单；完整覆盖由下游 PLAN 阶段对照 FACTS 原文完成。
- **并行 / 串行建议**：以方法论之间的依赖关系说明可以并行哪些、必须等待哪些，而不是逐条列子任务名。
- **向下游的交接**：本章末尾点一句"具体对象的逐项覆盖、具体输入与参数、具体依赖与排期由下游 PLAN 阶段基于 FACTS 完成"。

### 3. 验收标准 (Acceptance Criteria)

本章是**整份文档的硬约束**，下游 PLAN 与每个子任务的执行都必须满足。包含三个子节：

#### 3.1 覆盖率（Coverage）

- **FACTS 是覆盖率核验的唯一真相源**：下游 PLAN 必须直接以上游 FACTS 区块的原文为基准逐条对照，每一个 bullet 都要有至少一个 subtask 或 sub_subtask 引用。**本指导文档不负责、也不得提供清单**；重复提供只会因概括偏差造成漏项。
- FACTS 与 EVIDENCE 中出现过的**每一个待处理对象**（URL、文件、模块、函数、记录、配置项、组件、接口契约等任意形态），都必须被至少一个子任务显式覆盖。
- **量化约束**：若 FACTS 列出 N 个对象，子任务累计必须覆盖全部 N 个，任一缺失即视为验收不通过。
- **清单型 FACTS 严禁遗漏**：当 FACTS 以清单形式列出一组同类对象（URL 清单、文件清单、表清单等），下游 PLAN 必须逐条对照核验，不允许以"代表性样本"替代全量；特别当 FACTS 含 URL 清单时，下游 PLAN 必须覆盖清单中的每一条 URL，一条都不能漏。
- 本小节的输出只声明上述规则，**禁止在这里再把 FACTS 中的对象列一遍**（参见通用写作约束）。

#### 3.2 有效性（Effectiveness）

- 以**产出对目标问题的直接证据**为首要目标，而不是"走完了多少步骤"。
- 每一条结论必须同时包含：
  - **正向验证**：在预期条件下，可观测到的、能证明结论成立的信号
  - **负向 / 对照验证**：在对照基线或否定条件下的响应，用以排除误报、建立判定基线
- 每一条可观测结论必须附带关键**执行证据**：执行输入、产生的输出、关键日志 / 响应片段；对安全类任务尤其需要保留触发 payload 与完整请求响应等可复现字段，具体字段随领域而定。

#### 3.3 报告产出（Reporting）

- 最终交付必须为每一条"已确认结论"保留可复现的关键证据字段，字段含义由领域决定，建议遵循以下映射（按当前任务真实情况取用）：
  - 安全 / 渗透类任务：目标、风险类型、严重度、触发输入（payload）、请求与响应
  - 代码 / 变更类任务：变更对象、变更内容、变更前后差异、验证输出
  - 分析 / 诊断类任务：被分析对象、分析方法、关键观察、结论与置信度
- 未被证据支撑的结论**不得**写入最终报告；被证伪的假设应作为"否定结论"如实记录。

### 4. 动态重规划与纠错纠偏 (Replanning & Correction)

- **触发条件**：EVIDENCE 与 FACTS 出现冲突 / 目标不可达 / 执行受阻（拒绝、权限、超时、限流、资源不足等）/ 发现原计划未覆盖的新对象 / 子任务连续失败 / 某条方法论范式被证伪。
- **响应策略**：
  - **局部重规划**：仅调整受影响的子分支，保留其他路径继续推进
  - **全局重规划**：当目标或边界发生变化、或根基性假设被推翻时，推翻现有计划重建
- **纠偏动作**：补充侦查 / 补充阅读 / 回退到上一步 / 切换方法论范式 / 降级为人工确认 / 将新发现对象合入 FACTS 后重新走覆盖率校验。
- **证据回写**：任何重规划与纠偏决策都必须以 EVIDENCE 条目记录（触发原因、决策变更、影响范围），禁止只改计划不留痕。
