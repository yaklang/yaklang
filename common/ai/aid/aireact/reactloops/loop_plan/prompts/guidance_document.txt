# 输出要求

请按照以下结构输出一篇 Markdown 格式的任务执行指导性文档。每个章节都必须基于上方提供的事实和证据，禁止编造不存在于上下文中的信息。

## 文档结构

### 1. 目标定义 (Goal Definition)
- 对用户原始需求进行精确重述
- 定义可衡量的成功标准（交付物、验收条件）
- 明确任务的边界：做什么、不做什么

### 2. 系统现状分析 (Current State Analysis)
- 基于 FACTS 对当前系统/环境/项目进行客观描述
- 列出关键约束条件和边界条件
- 标注已知的技术栈、文件结构、依赖关系等

### 3. 差距分析 (Gap Analysis)
- 现状与目标之间的具体差距
- 需要修改、新增或删除的具体方面
- 识别关键风险和不确定因素

### 4. 控制策略 (Control Strategy)
- 实施的总体方向和方法论
- 技术路线选择及其依据
- 关键决策点和判断标准

### 5. 执行路径 (Execution Path)
- **逐项枚举**所有需要操作的目标（文件/端点/模块/组件），禁止用"等"概括
- 对每个目标列出具体的操作内容（修改什么、测试什么、验证什么）
- 变更范围的明确界定：哪些在范围内，哪些明确排除
- 执行顺序的建议（哪些可以并行，哪些必须串行），标注依赖关系
- 如果 FACTS 中发现了 N 个目标/端点/参数，执行路径必须覆盖全部 N 个，不能遗漏

### 6. 验证与反馈 (Verification & Feedback)
- **验证矩阵**：对每个目标 × 每种验证方法，列出具体的验证项
  - 安全测试场景：每个端点要测试哪些漏洞类型，用什么 Payload，期望什么响应
  - 开发任务场景：每个修改要通过哪些测试用例，预期输出是什么
- 每个验证项的**成功信号**（具体的响应内容、状态码、行为变化）
- 每个验证项的**失败信号**（错误信息、异常行为、超时）
- 整体完成判定标准：全部验证项通过的具体条件
- 回滚和纠错机制
