# Web API 文档编写规范（DOC_AUTHORING_SPEC）

本规范定义 Yaklang Web API 文档（`docs/api`）的"原材料"——即 `common/yak/yaklib/`（以及其它导出库）中 Go 函数注释——应当如何编写，才能被文档生成器（`generate_web_doc` + `webdoc` 渲染包）解析为高质量、结构完整、可校验的页面。

核心原则：**原材料即 Go doc 注释**。要丰满某个库的文档，只需按本规范改写其函数的 Go 注释；渲染、分组、校验全部由 `webdoc` 包完成，无需改引擎。

首批标杆库：`yakit`、`db`。新补/优化文档时，请以本规范为准绳，并参考这两个库的实际注释。

---

## 1. 注释结构契约

一个函数的 doc 注释（即紧贴 `func` 之上的 `//` 注释块）按以下顺序组织，各段均为可选但**强烈建议齐全**：

```go
// 函数名 一句话描述（导出名为 lib.Name）
//
// 这里写详细描述：用途、典型场景、注意事项、与相邻函数的关系等。可多段。
//
// 参数:
//   - paramName: 参数说明（与签名参数同名，逐个解释）
//   - other: ...
//
// 返回值:
//   - 返回值说明（按签名返回值顺序逐个列出）
//   - ...
//
// <|EXAMPLE_START|> 基础用法
// ```
// // 这里是 yak 代码
// lib.Func("x")
// ```
// <|EXAMPLE_END|>
//
// <|EXAMPLE_START|> 进阶用法
// ```
// // 另一个示例
// ```
// <|EXAMPLE_END|>
func Func(...) ... { ... }
```

解析规则（由 `webdoc` 实现）：

- **一句话描述**：注释第一行，进入"函数索引"表的说明列。
- **详细描述（功能）**：从注释开头到第一个结构标记（`参数:` / `返回值:` / `Example:` / `<|EXAMPLE_START|>`）之间的正文，渲染为函数详情的功能描述（自动去重，不会把参数/返回值列表重复 dump）。
- **参数说明**：`参数:` 段下的 `- name: 说明` 列表，按参数名匹配到签名参数，渲染进"参数 / 必填参数"表的说明列。
- **返回值说明**：`返回值:` 段下的 `- 说明` 列表，按顺序匹配签名返回值，渲染进"返回值"表的说明列。
- **示例**：见第 2 节。

约束：

- 注释里的 ``` 代码围栏必须成对（生成器 `CheckDocCodeBlockMatched` 会在不成对时直接 panic）。
- 字符串、log 输出一律英文；中文仅出现在注释里（遵循仓库规范）。

## 2. 多示例标记（EXAMPLE 标记）

一个函数可以有多个示例，每个示例用成对标记包裹，标题可选：

```
// <|EXAMPLE_START|> 可选标题
// ```
// ... yak 代码 ...
// ```
// <|EXAMPLE_END|>
```

- **每个示例都应写语义化子标题**（一句话描述这个示例在演示什么），如 `SetKey 的基础读写`、`带标签入库`。渲染为 `**示例：标题**`。
- 无标题时退化：单个渲染为 `**示例**`，多个渲染为 `**示例 1 / 2 / ...**`（不推荐，请尽量给标题）。
- 每个示例独立用 14 反引号 yak 围栏输出，便于文档站稳定渲染与抽取校验。
- 标记行内的 ``` 围栏会被自动剥离，最终统一改用 14 反引号围栏，因此示例正文里写普通三反引号即可。
- **向后兼容**：未使用任何 `<|EXAMPLE_START|>` 时，回退到旧的单 `Example:` 段；此时**只抽取 `Example:` 之后第一个 ``` 围栏块的内部**，围栏闭合后的 `参数:`/`返回值:`/补充说明等散文不会混入示例（否则中文会进入 yak 代码导致语法检查失败）。新写文档优先使用标记，便于增删多个示例。
- 目标是**增大示例容量**：核心函数给 2~3 个分级示例（基础 / 进阶 / 选项组合）；主函数主干（签名/必填参数/返回值）保持最稳定，示例放最后，增删互不影响。

**示例正文必须是合法 yak（会被强校验，见第 7 节）**：

- 解释性中文只能写成 yak 行注释（`// 说明`），不能作为裸句子单独成行。注意：Go doc 的 `// ` 前缀会被剥一层，所以源码里要写 `// // 说明` 才能在示例里渲染成 `// 说明`；或干脆把该句拆成独立的带标题 `<|EXAMPLE_START|>` 块。
- 不要写省略占位：单独成行的 `...`/`…` 会被自动丢弃，但请勿依赖它；要么写完整可运行片段，要么用注释说明省略。
- 用 yak 语法而非 Go：函数字面量用 `func(a, b){}`（**不带类型**），不要 `func(w http.ResponseWriter, ...)`；不支持 Go 复合字面量 `&pkg.Type{...}`（请改用库提供的构造/生成函数）；`make(chan T)` 的元素类型用 `any` 等 yak 可识别类型；正则等含 `\d` 的字符串用反引号原始串 `` `\d` `` 而非双引号。
- 避免用保留字当变量名（如 `fn`）；调用括号要配平（常见错误是多一个 `)`）。

主函数核心稳定性：**签名、必填参数、返回值这三段恒定在固定位置；示例放最后**，因此增删示例不会影响主干结构。

## 3. 选项（Option）关联规则

选项模式：某"主函数"带一个可变参数 `...XxxOption` 或 `...XxxOptions`，库内另有若干"选项函数"——其唯一返回值类型恰为该选项类型。例如：

- 主函数：`db.SaveHTTPFlowFromRawWithOption(url, req, rsp, exOption ...yakit.CreateHTTPFlowOptions) error`
- 选项函数：`db.saveHTTPFlowWithTags(tags string) yakit.CreateHTTPFlowOptions`

渲染策略（由 `webdoc.buildOptionIndex` 实现）：

- 选项函数从顶层"函数索引 / 函数详情"中**剔除**（不在开头单独列出）。
- 在**每个**消费该选项类型的主函数详情下，渲染 `**可选参数 / 选项**` 小节，逐条列出对应选项函数（库名.函数名 + 一句话描述 + 内联签名）。
- 多个主函数共享同一选项类型时，**重复展示**（在每个主函数下都出现）。

判定要点：

- 选项类型必须以 `Option` 或 `Options` 结尾，且库内确有"消费它的可变参数"，才会被识别（避免把 `...string`、`...interface{}` 误判）。
- 选项类型按"基名"匹配（去包限定、去前导 `*`），以兼容跨包限定差异：消费侧形参常带包名（如 `...fp.ConfigOption`），而生产侧函数定义在该包内、其返回类型记录为裸名（`ConfigOption`），二者本是同一类型，按基名比较才能关联。`servicescan`/`synscan` 的几十个 `fp.With*`/`synscanx.With*` 选项正是靠这条规则归到主函数 `Scan` 下。

因此，要让某选项出现在主函数下，只需：让选项函数返回该选项类型、主函数以 `...该类型` 作为可变参数，并给选项函数写好一句话描述。

## 4. 页面分组：两套索引 + 两块详情

每个库页面按"函数是否含可变参数（`...T`）"分成两组，**各有独立索引和详情块**：

- `## 函数索引` + `## 函数详情`：不含可变参数的普通函数。
- `## 可变参数函数索引` + `## 可变参数函数详情`：含可变参数的函数（包括 `yakit.Info(tmp, items ...any)` 这类普通可变，以及 `db.SaveHTTPFlowFromRawWithOption(..., ...Options)` 这类选项型可变）。

某一组为空时该组的标题与表格不渲染。锚点在"普通在前、可变在后"的合并顺序上统一分配，保证全页唯一、索引链接与标题一致。选项生产者函数不进任何索引（见第 3 节，只在消费它的主函数下展示）。

## 5. 函数详情叙事顺序（核心稳定）

每个函数详情按固定顺序渲染，主干位置恒定，便于核心稳定、示例自由增删：

1. 标题（带显式唯一锚点 `{#anchor}`）
2. 签名（go 代码块）
3. 功能描述（已自动去前导函数名）
4. 参数：
   - 含可变参数时：**必填参数**（非可变参数）+ **可选参数**（可变参数；其中选项型可变参数列出其全部选项函数）。
   - 不含可变参数时：**参数**（全部参数）。
5. **返回值**
6. **示例**（可多个，每个带语义化子标题）
7. 分隔线 `---`

即可变参数主函数脉络为：**主函数功能 - 必填参数 - 可变参数可用列表 - 返回值 - N 个带标题示例**。

## 6. 模块总览

每个库可有一段"模块总览"，讲清模块用途、典型场景、与相邻库的关系：

- 文件位置：`common/yak/yakdoc/generate_web_doc/overviews/<libname>.md`。
- 生成器通过 `-overviews <dir>` 加载（默认即该目录），渲染时注入到库标题 H1 之后、概览统计行之前。
- 纯 Markdown 库（如 `yakit`、`db`）的总览支持多段落、列表、行内代码（勿写裸 `<` 或裸 URL，遵循第 6 节不变量）。
- MDX 库（如 `ai`）的总览同时用于 frontmatter `description`，会被压成单行。

## 7. 校验流程（务必本地先过）

文档质量靠两层校验守护，二者都在 `essential-tests` 的 `./common/yak/yakdoc/...` 用例里运行（无需新开容器）：

### 7.1 结构不变量（`webdoc.CheckMarkdownInvariants`）

对每个生成的 `.md` 校验：代码围栏成对（含 14 反引号示例围栏成对）、仅一个 H1、标题不超过 3 级、显式锚点唯一且站内链接都有目标、同表各行列数一致、非代码处无未中和裸 URL、非代码处无裸 `<`。生成器写文件后会自检并打 error log。

### 7.2 示例语法校验（`webdoc.ExtractYakExamples` + antlr）

从生成产物抽取每个 14 反引号 yak 示例，交由引擎 `antlr4yak.New().FormattedAndSyntaxChecking` 做语法/编译检查（进程内、不联网、不执行）。**现已对【全部】导出库强约束**：`TestAllLibsExampleSyntax`（见 `example_syntax_test.go`）遍历所有库的所有示例，任何一个语法错误即测试失败。另有 `TestBenchmarkLibsHaveExamples` 要求标杆库（`yakit`/`db`/`servicescan`/`synscan`）至少含一个示例。

度量指标：**全库示例语法失败数必须为 0**。新增/修改示例后跑该测试即可量化（失败会打印库名、示例序号与报错代码）。

### 7.3 示例执行校验 + 本地 MOCK + "无法本地验证"标注

语法正确只是下限，**能跑通**才是更强的质量保证。执行校验位于 `example_exec_test.go`：

- `TestSafeLibsExampleExecution`（强约束，失败即红）：对"安全执行白名单 `safeExecLibs`"里的库，把每条示例用 `yaklang.New().SafeEval`（带 8s 超时、panic 恢复）真实执行。白名单只放**纯计算/本地解析、不出网、不执行命令、不写删文件、不需特权**的库（如 `codec/str/re/json/yaml/xml/regen/jwt/diff/xpath/java/jsonstream/env/log/memeditor/xhtml/sandbox` 等）。执行前 `setupLocalExecEnv` 会用**临时数据库**（`consts.InitializeYakitDatabase` 指向 `t.TempDir()`）隔离，杜绝读写用户真实 Yakit 数据；并起一个**本地 mock HTTP 服务**，地址写入环境变量 `YAK_DOC_MOCK_HTTP` 供示例联调。
- `TestAllLibsExampleVerifiabilityReport`（静态分类，不执行、不出网）：对全部库统计示例的"本地可验证性"，产出 `/tmp/doc_unverified.txt`（既不在白名单、也未标注的示例清单），用于持续推进覆盖。

**为什么不对所有库直接执行**：非纯计算库的示例会真实出网（已观测到 `bot` webhook 真的发出去）、可能执行命令/写删文件，甚至调用 `os.Exit`/`log.Fatal` 直接把测试进程打崩，违背"尽量不出网、无副作用"。因此对它们只做静态分类，把清单交给人逐个 mock 或标注。

**MOCK 优先，做不到再标注（强制要求）**：

- **能本地 mock 的，必须 mock**：涉及网络 IO 的示例应写成**自给自足、不出网**的形式——用 `tcp.MockServe(rsp)` / `httpserver.Serve(...)` 在示例内起本地服务，再请求它；或用 harness 预置的临时 DB / `YAK_DOC_MOCK_HTTP`。这样示例可纳入执行校验、可复制即跑。
- **确实无法本地验证的，必须显式标注**：在示例代码里写一行注释，包含短语 **`无法本地验证`** 并说明原因，例如：

```yak
// 无法本地验证: 需要真实可达的外部 HTTP 目标
rsp = poc.Get("http://example.com")~
```

  该短语既是给人读的说明，也是给执行校验用的机器标记：执行校验会**跳过**这类示例并计入"已标注"，便于后续单独处理（真机/真服务）。适用场景：真实外部目标/凭据、磁盘上需预置的真实文件（jar/zip/目录）、root 特权（如 syn 扫描）、第三方在线服务（AI/dnslog）等。

**度量指标**：(1) 白名单库执行失败数必须为 0；(2) 全库未验证示例数（`/tmp/doc_unverified.txt` 行数）持续下降——途径是把更多纯库纳入白名单、把网络示例改写成本地 mock、把确实跑不了的标注 `无法本地验证`。

### 7.4 本地检查清单（每次补/改文档前后执行）

```bash
# 1. 单元 + 边界 + 不变量 + 全库示例语法强校验 + 白名单库示例执行强校验
go test ./common/yak/yakdoc/...

# 2. 全库本地可验证性度量(静态, 不出网), 产出 /tmp/doc_unverified.txt
go test ./common/yak/yakdoc/generate_web_doc/ -run TestAllLibsExampleVerifiabilityReport -v

# 3. 全量重生成（可选，确认所有库都过不变量自检，观察是否有 error log）
go run -gcflags=all=-l ./common/yak/yakdoc/generate_web_doc /tmp/webdoc_out

# 4. 抽查目标库产物
#   - 索引/详情结构、必填参数/可选参数(选项)/返回值/示例 是否符合预期
#   - 模块总览是否注入
```

新增/优化文档的标准动作：改 Go 注释 → 如涉及总览则改 `overviews/<lib>.md` → 跑上面清单全绿 → 重生成预览。

## 8. 内容分级要求

- **核心函数（极致）**：丰富的功能描述 + 必填/可选参数（含选项）+ 多个分级示例（基础 / 进阶 / 选项组合）+ 返回值说明。示例尽量自包含、可独立通过语法检查，取材自本仓库 `*_test.go`、`yaklang.github.io` 现有文档、`yaklang-ai-training-materials/library-usage`。
- **其余函数（结构完整）**：至少 `一句话描述 + 参数: + 返回值: + 1 个示例`，且示例过语法检查。

参考实现：`common/yak/yaklib/db.go`、`common/yak/yaklib/yakit.go`。
