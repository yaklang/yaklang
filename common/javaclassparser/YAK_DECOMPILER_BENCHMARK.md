## 1. Bottom line: not GA yet

The decompiler is **safe and complete on its own synthetic corpus**, but **not
GA for real-world bytecode**. Two blockers stand between the current state and GA:

1. **A live P0 regression on this branch.** The latest committed change to
   `CalcOpcodeStackInfo` (tolerating empty if-merge sources by substituting an
   empty stack entry) sends `mergeIf` into an effectively non-terminating /
   pathologically slow loop on at least one regression class. Concretely,
   `TestGAPanicFreeBoundary/panic_nilref_typeutils` times out at 12-20s instead
   of returning. This breaks the 30s fast gate and any `.m2` sweep that hits a
   class of the same family. **This must be fixed (or reverted) before anything
   else.** Section 3 documents it precisely with the reproduction command.
2. **Residual real-jar partials (~0.4%).** On the real-world `.m2` corpus the
   decompiler still degrades ~0.4% of classes to a tagged stub. The synthetic
   corpus is at **0 stubs / 0 round-trip failures**; the gap is entirely in
   real-world control-flow structuring (loop/value-merge families). GA requires
   driving this toward zero, or an explicit, tested degradation policy.

Everything else - syntax safety contract, determinism, test hygiene, the
performance profile - is at a shippable, engineering-beta level today.

## 3. P0 regression on this branch (blocks GA)

**Symptom.** `TestGAPanicFreeBoundary/panic_nilref_typeutils` does not return -
it times out (reproduced at 12.7s and 20s `-timeout` settings). The other six
boundary classes in the same suite pass in 0.6-5.8s each. A full-suite
`go test ./common/javaclassparser/...` therefore hits the 2m test timeout at
this subtest.

**Root cause (localized).** The stack trace pins the stall inside
`(*RewriteManager).mergeIf` (`statement_wrap.go:345`), reached from
`ParseByteCode`. The committed change in `CalcOpcodeStackInfo`
("tolerate empty if-merge sources without erroring") removed the early
`return nil, errors.New("invalid if merge node")` guard and, for an empty
`validSources`, installs a fresh empty `StackSimulation`. For the
`typeutils` class this feeds `mergeIf` an input shape it descends into without
converging - a structuring-time blow-up, not a parser defect.

**Reproduce:**
```
go test -run 'TestGAPanicFreeBoundary/panic_nilref_typeutils' \
  -count=1 -timeout 20s ./common/javaclassparser/tests/
# panic: test timed out after 20s
```

**Effect on `.m2` sweeps.** Any real-jar scan that reaches a class of this
family stalls the single-goroutine scanner, so the `.m2` partial counts cannot
be reliably re-measured on HEAD until this is fixed. The baseline `.m2`
numbers in section 5 are therefore the last clean pre-regression snapshot.

**Fix path.** Either revert the empty-source tolerance and restore the error
(so the class degrades to a tagged stub via the existing safety net), or bound
`mergeIf` with a visit/iteration guard so an un-converging merge degrades
instead of spinning. Either preserves the safety contract; the former is the
minimal, safe, GA-unblocking move.

## 5. Real-world corpus (.m2) - last clean baseline

The `.m2` validation runs are heavy (per-class ANTLR re-parse, single goroutine,
~30 GB RSS at 6k classes), so they are run as periodic snapshots, not in the
fast gate. The last clean snapshot **before the section 3 regression**:

| Scan | Classes | ok | partial | syntax | err | panic |
|------|--------:|---:|-------:|-------:|----:|------:|
| 120-jar window | 12000 | 11965 | 35 | 0 | 0 | 0 |
| full `~/.m2` (`M2_INDUSTRY=1`) | ~94000 scanned | - | ~368 (~0.4%) | 0 | 0 | 0 |

GA target for real-jar coverage: drive `partial` toward 0, with every residual
degradation an explicitly tagged, still-parseable `yak-decompiler:` stub (the
safety contract already enforces this for terminating inputs).

### 5.1 The four residual structural families

The ~0.4% real-jar partials cluster into four families, all rooted in
loop / value-merge structuring rather than per-class quirks:

1. **Cyclic / shared-container statement** - `AssertStatementsAcyclic` rejects a
   container reachable from two parents; a post-loop `IfStatement` is
   double-attached both at loop tail and inside a nested `do-while` body
   (druid `TDDLHint`, jackson `UTF8DataInputJsonParser`).
2. **variable-fold nil ref key** - a `loadVarBySlot` of an uninitialized slot
   yields a nil `*JavaRef` that becomes a `varUserMap` key (fastjson2
   `TypeUtils.doubleValue`).
3. **multiple next** - a node retains two `Next` edges after structuring, tied
   to `break`/`continue` at different loop levels (fastjson2 `seekLine`).
4. **post-decompile syntax validation** - residual `ConditionStatement`s inside
   a structured `IfStatement` body `IfRewriter` did not recurse into, plus
   loop-exit mis-attribution (druid `SchemaResolveVisitorFactory.resolve`).

All four reduce to **loop-structuring mishandling post-loop / cross-loop code
when the loop body has `break`/`continue`** - the decompiler's single hardest
structural problem. Each is currently caught by a safety net, so the contract
holds; clearing them needs loop-rewriter surgery to (a) never double-attach a
node and (b) keep post-loop code out of the loop body.

## 7. Performance profile (unchanged, still valid)

The allocation/throughput profile is independent of the section 3 regression and
remains as previously characterized:

- **GC-bound.** Core decompile ~215 ms / ~161 MB cumulative heap per 106-class
  jar; the post-decompile ANTLR re-parse adds ~+60% runtime / ~+42% bytes.
- **Tail-bound.** On byte-buddy (2845 classes) one 43 KB class is 26% of a cold
  pass; the top 1% of classes are 61%. Average-case tuning moves throughput
  modestly; the high-value target is the pathological tail.
- **Parallel scaling.** Near-linear to ~4 workers, peaks ~8 (3.6x), then
  GC-regression. Raising the multi-core ceiling needs allocation reduction.
- **Cross-parse ANTLR cache** deliberately not pulled: the pinned ANTLR Go
  runtime has no locking on DFA/`JStore` structures and decompilation runs
  parallel, so a shared validation DFA would data-race. Needs an ANTLR upgrade.
